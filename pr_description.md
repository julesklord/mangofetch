💡 **What:** Replaced synchronous `std::fs` functions (`remove_file`, `rename`, `metadata`) with their asynchronous counterparts (`tokio::fs`) inside the async `download_hls` method.
🎯 **Why:** The HLS downloader handles many file operations during cancelation, error reporting, and successful completion. Calling synchronous filesystem operations (like `std::fs::remove_file`) directly within an asynchronous `tokio` context blocks the executor thread until the I/O operation is complete. This prevents the runtime from scheduling or executing other concurrent async tasks, which increases latency and reduces the efficiency of the runtime. By migrating to `tokio::fs`, the blocking file operations are delegated to `tokio`'s dedicated blocking thread pool, freeing up the async executor threads to handle other active downloads efficiently.
📊 **Measured Improvement:** It's impractical to set up a robust benchmark specifically to measure the impact of file operations inside the `download_cancelled` and `download_failed` paths as these execute sporadically based on user interaction or network faults. The primary benefit lies in better CPU scheduling for `tokio`'s async runtime by not blocking the executor thread during the I/O.
